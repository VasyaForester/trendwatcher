# TrendWatcher — план разработки агента мониторинга трендов AI Security

## 1. Цель и продукт

Агент непрерывно собирает публикации по AI security (и смежному AI-контексту), находит в них количественные и качественные паттерны и превращает их в:

1. **Веб-страницу** с дашбордами: топ событий, динамика ключевых слов, сигналы (weak/emerging/strong), тепловые карты по поверхностям атак, подборки материалов.
2. **PDF-отчет по требованию** — подробный квартальный (за прошедшие 3 месяца) trend brief с цитатами, ссылками и уровнями уверенности (по методологии skill `ai-security-trends-analysis`).

Ключевой принцип: **каждый тренд подкреплен ссылками на первичные источники**; сигналы без подтверждения помечаются как Unverified и не попадают в топ.

---

## 2. Источники данных

| Категория | Примеры источников | Способ сбора |
|---|---|---|
| Инциденты | AI Incident Database (AIID), AIAAIC, OECD AI Incidents Monitor, новости об инцидентах | API / RSS / скрейпинг |
| Уязвимости | NVD/CVE (фильтр по AI-продуктам: Ollama, LangChain, vLLM, MLflow…), GitHub Security Advisories, Protect AI / huntr | NVD API 2.0, GHSA GraphQL |
| Исследования | arXiv (cs.CR, cs.AI, cs.CL, cs.LG), Semantic Scholar | arXiv API, S2 API |
| Стандарты и фреймворки | NIST (AI RMF, GenAI Profile), OWASP (LLM Top 10, AI Exchange), MITRE ATLAS, ISO/IEC 42001, ETSI | RSS / периодическая проверка страниц + diff |
| Регуляторика | EU AI Act (имплементация), CISA/NCSC/ENISA/BSI advisories, FTC, российские регуляторы (по необходимости) | RSS / official feeds |
| Вендоры и Big Tech | Блоги OpenAI, Anthropic, Google DeepMind/Security, Microsoft MSRC, Meta, NVIDIA; security-релизы и model cards | RSS / блог-фиды |
| Инструменты | GitHub trending / releases по темам (guardrails, llm-security, red-teaming), PyPI-релизы ключевых пакетов | GitHub API |
| Новости и сообщество | The Hacker News, BleepingComputer, Simon Willison, специализированные рассылки; Hacker News (Algolia API) | RSS / API |

Новости — только как «lead»: тренд подтверждается первичным источником. Конфиг источников — декларативный YAML (url, тип, парсер, вес доверия, периодичность), чтобы добавлять источники без кода.

## 3. Архитектура пайплайна

```
[Коннекторы источников] → [Ingestion + дедупликация] → [Нормализация]
      → [Обогащение (LLM + NLP)] → [Хранилище: Postgres + pgvector]
      → [Аналитика трендов (batch, ежедневно/еженедельно)]
      → [API (FastAPI)] → [Веб-дашборды] + [Генератор PDF]
```

### 3.1 Ingestion
- Планировщик (APScheduler; при росте — Celery/beat): каждый источник по своему расписанию (новости — часы, стандарты — сутки).
- Дедупликация: канонизация URL + minhash/SimHash по тексту + косинус по эмбеддингам (кросс-источниковые дубли одного события связываются в «событие», а не удаляются).
- Сырые документы сохраняются (raw HTML/JSON) для воспроизводимости.

### 3.2 Нормализация
Единая схема документа: `id, source, source_type, url, title, text, published_at, fetched_at, language, authors, raw_ref`.

### 3.3 Обогащение (ключевой этап)
Для каждого документа LLM-пайплайн извлекает структурированный JSON:
- **Тип**: incident / vulnerability / research / framework / regulation / news / tool_release.
- **Таксономия AI security** (фиксированный словарь на базе OWASP LLM Top 10 + MITRE ATLAS + NIST AI RMF): prompt injection, jailbreak, data poisoning, model supply chain, agent/tool abuse, RAG poisoning, exfiltration, deepfake/fraud, governance и т.д.
- **Сущности**: вендоры, продукты, модели, CVE-ID, регуляторы, страны.
- **Атомарные утверждения (claims)** с привязкой к источнику — фундамент для claim ledger в PDF.
- **Флаги**: observed-in-the-wild vs proof-of-concept; severity (для инцидентов/CVE).
- Эмбеддинг текста (для кластеризации и семантического поиска).

### 3.4 Хранилище
- **PostgreSQL + pgvector**: documents, events (связанные документы об одном событии), claims, entities, taxonomy_tags, time_series (агрегаты), signals, reports.
- Материализованные представления для дашбордов (счетчики по неделям × тегам × типам источников).

## 4. Аналитика трендов

### 4.1 Количественные паттерны
- **Динамика ключевых слов/тегов**: частоты по неделям, скользящее среднее, velocity (Δ% period-over-period).
- **Burst detection** (алгоритм Клейнберга или z-score относительно скользящего baseline) — всплески упоминаний.
- **Novelty score**: термины/кластеры, отсутствовавшие в предыдущем окне.
- **Кросс-источниковая корроборация**: сигнал сильнее, если тема одновременно растет в ≥2 независимых типах источников (например, research + incidents, или CVE + vendor advisories). Это главный фильтр от хайпа.
- **Метрики по сущностям**: топ атакуемых продуктов/моделей, рост CVE по конкретным AI-инструментам.

### 4.2 Качественные паттерны
- **Кластеризация эмбеддингов** (HDBSCAN/UMAP) по скользящему окну → LLM именует кластеры и пишет 2–3 предложения «что это и почему растет».
- **LLM-синтез уровня «аналитик»** (еженедельно): what changed vs прошлый период, кто затронут, практический смысл, defensive recommendations — строго с цитатами из claims.
- **Классификация зрелости сигнала**: weak (единичные упоминания, 1 тип источника) → emerging (рост + 2 типа источников) → strong (устойчивый рост, corroborated, есть incidents/advisories) → mainstream/declining.

### 4.3 Скоринг «топ событий»
Композитный score события: severity × охват источников × доверие к источникам × новизна × скорость распространения. Топ-10 за неделю/месяц/квартал — первый блок дашборда.

## 5. Веб-страница

**Стек**: Next.js + TypeScript + Tailwind, графики — Recharts/ECharts. Backend — FastAPI (тот же, что кормит PDF-генератор). Данные обновляются batch-ом, поэтому фронт может быть почти статическим (ISR/revalidate).

Дашборды (MVP → полный набор):
1. **Top Events** — топ событий за период с score, типом, тегами, ссылками на первоисточники.
2. **Keyword / Topic Trends** — линии динамики выбранных тегов и терминов, переключение окна (4 нед / 13 нед / год), топ-движения (gainers/losers).
3. **Signals Radar** — weak/emerging/strong сигналы с обоснованием и уровнем уверенности.
4. **Attack Surface Heatmap** — матрица «поверхность атаки (по шаблону skill: model/prompt, RAG, agents, supply chain, deployment) × неделя», интенсивность = активность.
5. **Feed & Collections** — курируемые подборки (фильтры по типу/тегу/сущности), полнотекстовый + семантический поиск.
6. **Regulatory Tracker** — таймлайн регуляторных событий (EU AI Act milestones, NIST-публикации и пр.).
7. Кнопка **«Скачать PDF-отчет за 3 месяца»**.

---

## 6. PDF-отчет (квартальный trend brief)

Генерация строго по шаблону skill `ai-security-trends-analysis`:
- Executive summary (5–10 буллетов: тренд — почему важно — кто затронут — confidence).
- What materially changed this quarter.
- Threat landscape по поверхностям атак.
- Incidents & case studies.
- Defensive recommendations (prioritized).
- Metrics & signals to track; Open questions.
- Appendix: source list + claim ledger (claim → источники → confidence).

**Пайплайн генерации**: выборка данных за 90 дней → LLM-синтез по секциям (каждое утверждение обязано ссылаться на claim из БД — валидация на этапе генерации) → HTML-шаблон (Jinja2) → PDF через WeasyPrint или Playwright (print-to-pdf). Confidence по рубрике High/Medium/Low из skill. Отчеты кешируются и версионируются.

---

## 7. Технологический стек (сводно)

- **Язык**: Python 3.12.
- **Сбор**: httpx + feedparser + trafilatura (извлечение текста), tenacity (ретраи).
- **Обогащение**: LLM API (structured output / JSON mode) + эмбеддинги; scikit-learn/HDBSCAN для кластеризации.
- **БД**: PostgreSQL + pgvector (на старте допустим SQLite + sqlite-vec для локального MVP).
- **API**: FastAPI. **Планировщик**: APScheduler.
- **Frontend**: Next.js + Recharts. **PDF**: Jinja2 + WeasyPrint/Playwright.
- **Деплой**: Docker Compose (db, worker, api, web); мониторинг пайплайна — простые health-метрики (успешность сбора по источникам, лаг данных).

## 8. Этапы реализации

### Этап 0 — фундамент (неделя 1)
Схема БД, конфиг источников (YAML), скелет проекта, Docker Compose.

### Этап 1 — MVP пайплайна данных (недели 1–3)
5–7 источников (arXiv, NVD, AIID, 2–3 RSS новостей/блогов), ingestion + дедупликация + нормализация, LLM-обогащение с таксономией, наполнение БД. **Критерий**: 90 дней истории собрано, теги проставлены.

### Этап 2 — аналитика (недели 3–5)
Временные ряды тегов, burst detection, скоринг топ-событий, кластеризация + именование кластеров, классификация сигналов. **Критерий**: воспроизводимый weekly-отчет в JSON.

### Этап 3 — веб-дашборды (недели 5–7)
FastAPI-эндпоинты + Next.js: Top Events, Keyword Trends, Signals, Feed. **Критерий**: рабочая страница на реальных данных.

### Этап 4 — PDF-отчет (недели 7–8)
LLM-синтез по шаблону, claim-валидация, HTML→PDF, кнопка выгрузки. **Критерий**: квартальный отчет проходит ручную проверку на достоверность цитат.

### Этап 5 — расширение (после MVP)
Больше источников (GHSA, GitHub trending, регуляторы), Heatmap и Regulatory Tracker, семантический поиск, алерты (email/Telegram-дайджест), сравнение кварталов.

## 9. Риски и меры

| Риск | Мера |
|---|---|
| LLM-галлюцинации в синтезе | Каждое утверждение обязано ссылаться на claim с URL; валидация ссылок при генерации; Unverified не попадает в выводы |
| Хайп-искажение (новости раздувают тему) | Корроборация по ≥2 независимым типам источников; веса доверия источников |
| Ломкость скрейперов | Приоритет API/RSS; мониторинг успешности сбора; изоляция парсеров по источникам |
| Дрейф таксономии (новые типы атак) | Тег `other/emerging` + периодический пересмотр словаря по кластерам без тегов |
| Стоимость LLM-обогащения | Дешевая модель для рутинной разметки, сильная — только для еженедельного синтеза и PDF; кеширование |
| Юридические ограничения источников | Соблюдение ToS/robots.txt, хранение выдержек и ссылок, а не перепечатка полных текстов |

## 10. Метрики качества самого агента

- Покрытие: доля значимых событий квартала, попавших в систему (ручная выборочная проверка).
- Точность разметки: выборочный аудит тегов/типов (цель ≥90%).
- Достоверность отчета: 100% ключевых утверждений с рабочими ссылками.
- Свежесть: лаг «публикация → появление в системе» < 24 ч для новостных источников.
